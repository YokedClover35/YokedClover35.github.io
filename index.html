What Is A Sort?

Sorts are a way of arranging items in a container so that they follow some order. In the examples the containers used are a fixed length array with randomly generated integer values (whole numbers). Sorts can be used on all kinds of data though, some examples include dates, strings (text), numbers, etc...

What Makes A Sort Good?

Sorts that use less time or memory to complete are considered better. The way we measure how long or how much memory something will take to complete is by using big O notation of some variable (most commonly n). You can learn more about big O notation here. A simple definition of big O notation is a way to measure how fast the time/memory grows depending on the input size. Big O notation is by no means the only way to measure the quality of an algorithm. Some other metrics for sorts in particular include whether the sort is in-place (uses only the memory space it was given) and whether the sort is stable (does it keep equal items in the same order relative to each other). As a general rule of thumb O(1) < O(n) < O(nlogn) < O(n²) < ... remember the smaller the better.

What is the Difference Between Best, Avg, and Worst Case?

  • Best-Case: best-case occurs when your data is set up for your algorithm to do the least possible work. In the case of a sort that is most likely a container that is already in sorted order. best-case is the least significant out of the 3 cases because your algorithm is almost never going to encounter the best case in practice

  • Avg-Case: avg-case is short of average-case and this is the situation that your algorithm will encounter the most, it is the average of all possible data sets. This is very important for understanding how your algorithm will run when out in the wild.

  • Worst-Case: worst-case occurs when your data is set up to spite you, this represents the most work your algorithm will ever have to do so it is very important to keep in mind. Although an algorithm may have a better worst case runtime than another it is still important to keep in mind that practically speaking the average case is the most important. An example of this is the worst-case runtime of quicksort is greater than the worst-case run time of mergesort but in practice mergesort usually takes longer than quicksort.
